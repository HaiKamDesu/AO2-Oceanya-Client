# AO2 Viewport

## Purpose
The AO2 viewport is the GM multi-client visual surface that mirrors the original AO2 client courtroom viewport at the native 256x192 resolution.

## Entry Points
- `OceanyaClient/MainWindow.xaml`: `Main.Viewport.Open` (`V`) button left of the area navigator `A` button.
- `OceanyaClient/MainWindow.xaml.cs`: opens the viewport as an owned, non-taskbar `GenericOceanyaWindow`.
- `OceanyaClient/Features/Viewport/AO2ViewportWindowContent.xaml`: resizable hosted content that scales the native 256x296 viewport/chatbox surface with a centered, aspect-ratio-preserving `Viewbox`.
- `OceanyaClient/Features/Viewport/AO2ViewportControl.xaml`: AO2 layer stack.
- `OceanyaClient/Features/Viewport/AO2ViewportAssetResolver.cs`: AO2-style asset lookup for backgrounds, desks, character sprites, and effects.
- `AOBot-Testing/Structures/ICMessage.cs`: parsed incoming `MS#` chat packet fields consumed by the viewport.

## Window Ownership
The viewport is a separate WPF window for movement and closing, but it is owned by the main GM window and has `ShowInTaskbar=false`. This gives Windows the intended grouped behavior: it does not appear as a separate taskbar window, and it follows owner activation/minimize/close behavior.

`AO2ViewportWindowContent` keeps one `AO2ViewportControl` per GM profile. Selecting a GM client only changes which profile viewport is visible; it does not reuse one render surface for every profile. Hidden profile viewports stay attached to their incoming message source and continue their own timers/animations, so switching back to a profile shows that profile's current scene instead of a blank/background-only reset.

In single-internal-client GM mode, each profile viewport listens to the single live AO2 connection through profile filters. A server echo is routed to the profile whose planned INI puppet id or local character/showname matches the packet. If only one internal profile exists, this degenerates to the same single viewport behavior.

The popup size and position are saved in `SaveFile.Data.GMViewportWindowState`. Width and height are stored as the viewport content area, not the outer generic window size; restore applies the generic chrome offsets once when creating the host window. This matches `OceanyaWindowManager`, which keeps hosted content size and host window size synchronized separately. The default and minimum content surface is the native 256x296 viewport/chatbox size. The host window enforces the true OS-level minimum with `WM_GETMINMAXINFO` and edits active drag rectangles with `WM_SIZING`, so aspect locking happens during native resize instead of by correcting WPF size after layout. Extra backdrop exposed by the uniform scaler is transparent, letting the generic Oceanya lab background show through.

## Current Layer Order
The WPF layer order follows AO2 `Courtroom` construction:
1. background (ZIndex 1)
2. speedlines placeholder (ZIndex 1)
3. main character (ZIndex 4, or 3 when pair is in front)
4. paired character placeholder (ZIndex 3, or 4 when pair is in front)
5. desk (ZIndex 5)
6. character sticker (ZIndex 6) — `sticker/{characterName}` shown during speaking messages
7. effect (ZIndex 8, adjustable via `layer` metadata)
8. evidence overlay (ZIndex 9) — evidence icon/appear animation when MS# has evidence ID
9. shout/objection overlay (ZIndex 9)
10. flash overlay (ZIndex 10)
11. testimony overlay (ZIndex 11) — "testimony" image shown on RT# testimony1
12. WT/CE/verdict overlay (ZIndex 12) — shown on RT# testimony2/judgeruling
13. AO2 chatbox (below viewport, row 1)
14. chat arrow overlay (row 1 canvas, above chatbox) — shown after text crawl completes

`AO2ViewportControl` adjusts Z-order at render time for AO2 pair ordering and effect `layer` metadata.

The resolver reads the selected AO2 `config.ini` `theme`, `subtheme`, and `theme_scaling_factor` instead of guessing a preferred viewport theme. With the user's current `theme=(714x688) FullChar` and `subtheme=default`, default chatboxes therefore use that theme's `ao2_chatbox=0,178,256,104`, `showname=1,0,46,15`, `message=10,12,242,89`, and `chat_arrow=245,84,11,9` rather than generic `themes/default` or `misc/default` placement. Character custom chatboxes use `[Options] chat` through `AO2ViewportAssetResolver.ResolveCharacterChatToken(...)`; `misc/<chat>` art/config beats generic theme chatbox art. Config filenames are resolved case-insensitively to match AO2 on Windows. If a custom chatbox only supplies art, `courtroom_design.ini` and `courtroom_fonts.ini` fall back through the active theme chain, not unrelated preferred themes; `misc/default/config.ini` remains only a legacy color-markup fallback for `chat_config.ini`. Live viewport message text uses the incoming `MS#` packet text color, matching AO2's `filter_ic_text(..., TEXT_COLOR)` path, rather than the theme's default message color. Message text keeps a 4 px document margin to match Qt `QTextDocument` placement inside the AO2 `message` rectangle.
- `AO2-Client/src/options.cpp` and `AO2-Client/src/courtroom.cpp`: AO2 stores audio sliders as `default_music`, `default_sfx`, and `default_blip` in `config.ini` on a 0-100 scale. `music_player` uses the music slider for the main music channel and the SFX slider for the ambient channels; `sfx_player` uses the SFX slider; `blip_player` uses the blip slider. That split is the reference for the Oceanya audio settings UI.
- During message crawl, live viewport message text uses AO2-style IC formatting: packet text color index, `chat_config.ini` `c0..c8` colors, `cN_start`/`cN_end` inline color markup, removed markup tokens, `~~` center, `~>` right, `<>` justify, `{`/`}` speed markers, escaped `\n`, `\p`, `\s`, and `\f`. Visible color delimiters render using AO2 delimiter-side color behavior, so a closing delimiter remains in the active color before the color stack pops. `PreviewText` updates only the text document, caret, and scroll position, mirroring AO2's per-tick `setHtml` + `setTextCursor` + `ensureCursorVisible` path without re-resolving layout/images on every character. The native viewport chat crawl reads AO2's `text_crawl`, `blip_rate`, and `blank_blip` values from the active `config.ini`, uses AO2's integer-truncated delay math, and starts the first character through a zero-interval timer like AO2's `chat_tick_timer->start(0)`. It keeps AO2's one-character-per-tick behavior and schedules each following timer with exactly the computed AO2 `msg_delay`; it does not shorten later intervals to catch up for dispatcher latency. `stay_time` controls AO2's post-message queue hold and `chat_ratelimit` controls outgoing chat submission, so neither changes incoming viewport blip cadence.
- `OceanyaClient/Components/Ao2AnimationPreview.cs`: provides static previews and animation players. Viewport playback now requests full-fidelity GIF decoding instead of the preview sampler so long AO2 GIF preanimations such as `VydValkKam/Dmc4intro.gif` are not reduced to 180 preview frames.
- `OceanyaClient/Components/Ao2AnimationPreview.cs`: provides static previews and animation players. Viewport playback now requests full-fidelity GIF decoding instead of the preview sampler so long AO2 GIF preanimations such as `VydValkKam/Dmc4intro.gif` are not reduced to 180 preview frames. Animation playback is scheduled to the next actual frame boundary instead of being polled on a fixed interval, and late ticks now catch up against the original frame schedule instead of rescheduling from "now", which removes the accumulated slowdown that was dragging long emotes and their SFX behind AO2.
- `AOBot-Testing/Structures/CharacterAssetPathResolver.cs`: resolves character asset tokens with a case-insensitive fallback to match AO2/Windows behavior for packets whose animation token casing differs from filenames.
- `AOBot-Testing/Structures/ICMessage.cs`: parses AO2 `MS#` fields. `OtherCharIdRaw` preserves CCCC pair ordering suffixes such as `12^1`; `OtherOffsetVertical` preserves vertical pair offsets where present. Packet layout (Compact vs Legacy) is detected by field count: >= 29 fields uses Legacy (which is what all AO2 servers produce in their echo), < 29 uses Compact (our own outgoing format, max 28 fields). This threshold matters — tsuserverCC always echoes 30 fields in Legacy layout, and using the wrong layout silently maps feature fields (additive, screenshake, effects) to wrong indices.
- `UnitTests/StructureTests.cs`: unit coverage for resolver behavior and packet fields that the viewport depends on.

## Chat Data Flow
1. The network `AOClient` receives an `MS#` packet from the server and raises `OnICMessageReceived`. In single-internal-client mode this is `singleInternalClient`, not the selected GM profile object.
2. `AO2ViewportControl.OnICMessageReceived()` dispatches to the UI thread and calls `RenderMessage()`.
3. `RenderMessage()` resolves packet character/showname/background/side data, then follows AO2's message order:
   - show shout overlay first when `ShoutModifier` is Hold It, Objection, Take That, or Custom;
   - render the preanimation phase when AO2's normalized `EmoteModifier`/`Immediate` behavior requests it and the packet `PreAnim` asset exists;
   - render the speaking/chat phase from the same packet, not from currently selected UI widgets.
4. The chatbox is rendered below the viewport. This intentionally differs from stock AO2's overlay position to avoid covering the scene.
5. Background and desk are resolved from the packet side and current background. Character, pair, flip, offsets, effects, speedlines, and chatbox contents are derived from the same `ICMessage`.
5. Selecting another GM client profile reveals that profile's persistent viewport control. Character or emote UI changes still do not repaint the viewport by themselves; the next matching IC message supplies that profile's new visible state, matching AO2's message-owned courtroom update behavior.

## AO2 Reference Behavior
The C# implementation should be compared against these AO2 files before changing viewport behavior:
- `AO2-Client/src/courtroom.cpp`: `set_background`, `handle_ic_message`, `handle_emote_mod`, `play_preanim`, `set_scene`, `initialize_chatbox`
- `AO2-Client/src/courtroom.h`: viewport constants and layer fields
- `AO2-Client/src/animationlayer.*`: frame playback, scaling, flipping, playback-once, and layer composition
- `AO2-Client/src/path_functions.cpp`: `get_pos_path` for `design.ini` origin/background/desk resolution.
- `AO2-Client/src/text_file_functions.cpp`: `get_effect_property` for `effects.ini` metadata.

Important AO2 contracts:
- Native viewport size is 256x192.
- Background position aliases include `def`, `hld`, `jud`, `hlp`, `pro`, `wit`, `jur`, and `sea`.
- Legacy background filenames such as `defenseempty` and `prosecutorempty` remain valid.
- Desk visibility depends on packet `deskMod`. AO2 has no runtime `DESK_CHAT` branch; a literal incoming packet value of `chat` is parsed by Qt as `0`, so the desk/overlay is hidden. Oceanya mirrors that for receive-time viewport rendering and resolves local outgoing unset/chat-like desk state to numeric show/hide values before sending.
- Background and desk move/scale together when a background position has an AO2 design origin.
- Effects may be behind the character, under the desk, over the viewport, or in the chat overlay layer depending on effect metadata.
- Pair characters use packet pair id, pair emote, flip, offset, and `^order` metadata.
- Zoom and preanim-zoom hide the desk/pair and choose prosecution or defense speedlines from packet side.
- Objection/Hold It/Take That/Custom shouts gate the normal message path until the shout overlay finishes.
- AO2 does not apply the new chat packet scene before a shout finishes; it overlays the shout on the previous viewport scene, then continues into the same message's preanim/speaking path.
- AO2 uses `shout_static_time=724` for non-animated shout gates and caps animated shouts with `shout_max_time=1500`; Oceanya uses the same 724 ms default gate for built-in shout overlays.
- Shout overlay visuals are resolved only from configured AO roots (`Globals.BaseFolders`): character folder, active misc/theme folders, theme default, then AO base root. The viewport must not use bundled Oceanya installation/source assets for AO visuals.
- AO2 plays character shout SFX through a dedicated objection/shout player while using the regular SFX lookup (`holdit`, `objection`, `takethat`, or `custom`) at the same time the shout overlay starts. Oceanya keeps the same separate player so later emote/preanim SFX cannot cut the shout off.
- Preanimations use packet field `PRE_EMOTE` (`ICMessage.PreAnim`) directly. They are not derived from the selected emote's configured preanimation name during receive-time rendering.
- Character talking/idle sprites use packet field `EMOTE` as AO2's raw animation token. The resolver may use the emote table when the packet value matches an emote name/id, but it must fall back to resolving `(b)<packet emote>`, `(a)<packet emote>`, and raw packet token paths because AO2 sends and loads the animation token.
- Character and preanimation asset lookup is case-insensitive in practice because AO2 is normally running on Windows and because content often differs in casing, e.g. `preexplosionrev` packet token vs `PreExplosionRev.gif`.
- AO2 normalizes legacy emote modifiers in `handle_emote_mod`: `4` is treated as preanim zoom, `2` is treated as preanimation, and unknown values fall back to idle/no-preanim.
- AO2 `handle_emote_mod` sends `PREANIM`/`PREANIM_ZOOM` through `play_preanim(false)`, which blocks the chat/speaking phase until `text_delay_timer` or `finishedPreOrPostEmotePlayback`. `IDLE`/`ZOOM` with the immediate flag go through `play_preanim(true)`, then immediately enter `handle_ic_speaking`.
- In this AO2 submodule, `play_preanim(false)` reads `[Time]`, `[stay_time]`, and SFX delay from `char.ini`, but only `stay_time` and SFX delay are multiplied by `time_mod=40` and actively scheduled. The `[Time]` value is passed as a duration limit into `CharacterAnimationLayer::loadCharacterEmote`, but the duration timer is not started by the referenced code path, so animated preanims normally advance by their own playback completion. Oceanya mirrors that by waiting for the preanimation player's `PlaybackFinished` event; `[Time]` is only a static/fallback estimate now.
- During speaking, AO2 chooses the talking `(b)` sprite while chat text is active unless the packet text color disables talking; otherwise it chooses the idle `(a)` sprite. When `chat_tick()` reaches the end of the message, AO2 switches to a one-shot `(c)` post-emote if it exists, then returns to `(a)` idle. If no `(c)` asset exists, it returns directly to `(a)` idle.
- AO2 screenshake affects `ui_vp_background`, `ui_vp_player_char`, `ui_vp_sideplayer_char`, and `ui_vp_chatbox`, not one single whole-viewport transform. Each layer gets independent random offsets every 20 ms for 300 ms with max deviation `7 * viewport_height / 192`, then returns to its original position.
- AO2 realization calls `do_flash()`, which resolves the `realization` effect and its sound. Oceanya resolves the same effect path when present and falls back to a white flash overlay when no local realization visual is available, so the user-visible flash remains present.
- AO2 starts `chat_tick()` immediately for non-empty messages, reveals text by grapheme, treats `{`/`}` as speed controls, handles escaped `\n`, `\p`, `\s`, and `\f`, delays punctuation, and gates blips through `blip_rate`, `blank_blip`, and anti-spam logic derived from `text_crawl / msg_delay`. Blip token fallback is `[Options] blips`, then `[Options] gender`, then `male`, with per-emote `[OptionsN]`/`[OptionsX]` overrides. The Oceanya viewport now follows those config-driven text/blip rules and triggers `\s` screenshake during the crawl.
- AO2 IC formatting is not plain text. `filter_ic_text()` removes alignment and speed-control tokens from display, converts escaped newline to a soft line break, suppresses escaped pause/shake/flash tokens, applies packet text color as the default color index, and applies `chat_config.ini` color-markup stack rules while preserving non-removed markup tokens.
- Additive IC packets keep a viewport-local previous text buffer and prepend it to the next additive message during crawl, then clear it on the next non-additive message or room/background reset. This mirrors AO2's `additive_previous` behavior closely enough for visible chat continuity while still using Oceanya's below-viewport chatbox placement.
- AO2 sends a blank/custom showname fallback from the selected INI puppet, not necessarily the locally iniswapped render character. Oceanya mirrors this for outgoing IC packets so a profile using an iniswap such as `VydValkKam` on the `Franziska` server slot sends `Von Karma` when the user left the IC showname field blank.
- Server position state is driven by `SP#` packets and the outgoing `MS#` position field. An empty `SP#` is meaningful: tsuserver resets the server position to the selected character's default side on character change. Oceanya preserves that empty state internally and displays it as `default (<character side>)` in the position dropdown. The dropdown list comes from `Background.GetAo2PositionOptions()`: AO2 default positions, then `design.ini` declared positions, then undeclared image-backed positions.
- AO2's `get_showname(char, emote)` also applies `[OptionsN]` to `[OptionsX] showname` overrides. Oceanya parses those overrides and applies them to outgoing blank-showname fallback, so server-slot puppets such as `Franziska` can send emote-specific names like `von Karma` even when the local render folder is an iniswap such as `KamLoremaster`.
- AO2 clients send Compact IC packets (no OtherName/OtherEmote/OtherOffset/OtherFlip pair fields), but AO2 servers always add those pair fields when echoing. The echo is always in Legacy layout. `ICMessage.FromConsoleLine` uses field count to distinguish: >= 29 fields → Legacy, < 29 → Compact. Compact maxes at 28 fields (all features). tsuserverCC always echoes 30 fields (Legacy without Blips/Slide). This is critical: the wrong layout silently maps additive, screenshake, immediate, and effect string to the wrong indices.
- AO2 parses desk modifier fields with `QString::toInt()`: a non-empty non-numeric token, including legacy `chat`, becomes `0` (`DESK_HIDE`), while an empty/missing character INI desk field remains unspecified. Some character packs, including `KamLoremaster`, place a desk image path in that field; Oceanya intentionally treats that as hidden so outgoing packets do not incorrectly force the desk visible.
- Desk mod 5 (`DESK_PRE_ONLY_EX`) shows the desk during preanim, then hides desk/pair and centers the main character for the speaking phase.
- Chatbox font/image behavior is delegated to the existing AO2 chat preview resolver; placement is below the viewport in Oceanya.
- AO2 audio is volume-segmented: music, SFX, and blips are independently configurable and should only be heard in Oceanya when the viewport is actually visible. For parity work, the SFX bucket should include client chirps such as the bell/ding and feature-entry sounds, while viewport message playback should stay tied to the active incoming IC message stream rather than the selected GM UI state.
- The GM viewport button is a show/hide toggle. `SaveFile.Data.GMViewportWindowState.IsVisible` records the last visible state, so the viewport opens automatically on the next GM multi-client launch when it was left shown. This restore behavior is always enabled and is not exposed as a user setting.
- Callword rules live in `SaveFile.Data.CallwordRules`. `AO2 Callword` rules sync to `config.ini` `callwords` and use AO2's default notification SFX; Oceanya-only trigger rules can match message contains/starts-with, character speaks, showname speaks, or character+emote usage and can use a custom SFX or the same AO2 default notification path resolved from `word_call`. Text callword rules can opt into whole-word matching, which requires non-letter/digit/underscore boundaries around the match so `Kam` matches `Kam, hello` but not `Kamui` or `akam`. Extra audio rules live in `SaveFile.Data.ExtraAudioRules`; current UI supports blip volume overrides by local character folder, local showname, or local blip token; SFX overrides by local character folder, local showname, or local SFX token; and music overrides by local music token. Character rules use the reusable main-client INI selector plus database-viewer selector mode, showname rules store an optional case-sensitive match flag, and audio-token rules use editable local dropdowns plus preview when the file can be resolved. Blip previews use the same centralized crawl/blip-rate/blank-blip rules as the viewport with the test sentence `This is a test message to scroll through blips, hello!`, while SFX previews are one-shot and music previews are play/stop. Extra audio rule volume is an override of the normal blip/SFX/music slider for matching audio, not a multiplier; the slider provides 0-200% quick input while the numeric field accepts higher positive percentages. Settings also expose AO2 `suppress_audio` as inverse `Unfocused Volume`, where 100% writes `suppress_audio=0` and 0% writes `suppress_audio=100`.
- The current viewport audio path is still a lightweight local playback bridge rather than a byte-for-byte reimplementation of AO2's full courtroom audio stack. It should be good enough for volume and visibility parity, but there may still be differences in long music looping, ambient channel mixing, and any blip behavior that depends on AO2 rich-text/markdown filtering beyond the current viewport crawl subset. Custom blip tokens now also resolve AO2-style relative paths rooted from `sounds/general`, including `../blips/...` fallback and character-relative custom paths.

## Known Parity Gaps
- Preanimation timing now waits for animated playback completion first, matching AO2's receive-time `play_preanim(false)` behavior. `[stay_time]` fallback values use AO2's `time_mod=40`; `[Time]` remains only a static/unsupported-animation fallback, not a hard cutoff for normal animated GIF playback.
- Objection/shout overlays are displayed with animated frame playback when the asset format is supported, but their continuation timing is still approximated by a timer.
- Currently implemented viewport overlay/layer families: background, speedlines, main character, pair character, desk/position overlay, AO2 effect layer metadata, shout/objection bubble, legacy realization flash, below-viewport chatbox, testimony overlay (RT# testimony1), WT/CE/verdict overlays (RT# testimony2/judgeruling), character sticker, evidence presentation overlay, chat arrow, and background slide transitions. Remaining gaps: exact AO2 queue/drain behavior around these overlays; effects loop/cull/max-duration/scaling; character SFX during some edge-case animation paths; slide transitions for character sprites (only BG/desk animate); WtCE overlays fall back to a 1500ms timer when no asset is found rather than AO2's exact SplashAnimationLayer timing.
- Packet-level screenshake now matches AO2's main trigger points more closely: it respects `config.ini` `shake`, uses the AO2 300 ms / 20 ms shake window, triggers immediate shake only for AO2 idle/zoom speaking messages, triggers preanim shake at SFX time, and also responds to frame-triggered `_FrameScreenshake` entries.
- Frame-timed emote SFX and realization flashes are now driven from the resolved animation token and local/packet `_FrameSFX` / `_FrameRealization` data instead of only from the coarse packet `SoundT` delay, but a few edge-case animation paths still need parity work.
- Background, desk/background overlay, character, pair, effect, speedline, shout, testimony, wtce, sticker, and evidence images use the shared WPF animation player for GIF/APNG/WebP where supported. Viewport GIF playback bypasses preview frame caps. The player now re-arms itself at the next due frame time instead of polling on a coarse fixed interval.
- Chat arrow position is read from `chat_arrow` in `courtroom_design.ini` via `AO2ChatPreviewResolver` (default: FullChar theme value `245, 84, 11, 9`). `AO2ChatPreviewControl.GetChatArrowBounds()` exposes the resolved value; `ShowChatArrow` positions `ChatArrowImage` dynamically from it. Arrow art follows AO2 `InterfaceAnimationLayer::loadAndPlayAnimation("chat_arrow", current_misc)`: the current message's `[Options] chat` misc token is searched before theme/default art, and path casing is resolved case-insensitively like AO2 on Windows. This matters for packs such as Akihiko/P4 where `misc/p4/Chat_Arrow.*` is a 1x1 transparent arrow; Oceanya must pick that invisible custom arrow instead of falling through to the visible FullChar arrow.
- Chatbox lookup mirrors AO2 `initialize_chatbox()`: resolve `[Options] chat` as the misc token, apply `ao2_chatbox`, `showname`, and `message` from the selected `config.ini` theme/subtheme chain, load `chatblank` for blank shownames or `chat`/`chatbox` for named messages, then apply `showname_extra_width` with sibling `chatmed`/`chatbig` art when the rendered showname exceeds the configured showname bounds.
- All animated asset slots (background, desk, main character, pair character) offload their first-time decode to a background thread via `Task.Run` + `Dispatcher.BeginInvoke`, matching AO2's `AnimationLoader` / `QtConcurrent` pattern. A `CancellationTokenSource` per image slot (`pendingAsyncLoads`) cancels stale in-flight decodes when the scene changes. Cached assets and static images remain on the synchronous fast path (instant). For character sprites the pre-animation → speaking transition is driven through an `onPlayerReady` callback: `RenderPreAnimationThenSpeaking` builds `handlePreAnimationPlayer` before calling `RenderScene`, passes it to `SetCharacterAnimatedImageAsync`, and the callback fires either synchronously (cache hit) or on the dispatcher after the background decode completes — in both cases triggering the `PlaybackFinished` hook that advances to the speaking scene.
- AO2's full chat queue lifecycle is still approximated: incoming messages replace the current viewport state instead of waiting on exact text completion and queue drain conditions.

## Image Scaling Pitfalls

### BitmapScalingMode
AO2 uses Qt `SmoothTransformation` (bilinear) for all image scaling. The WPF equivalent is `BitmapScalingMode.Linear`. `BitmapScalingMode.HighQuality` uses Fant resampling, which is noticeably softer than bilinear and must not be used as the default. The mapping in `ResolveDisplayOptions` is authoritative.

Character sprite layers follow AO2 `AnimationLayer::calculateFrameGeometry`: non-stretched character frames scale to the viewport height, and over-wide frames are clipped by the 256x192 canvas instead of being width-constrained. In WPF this is handled by `AO2ViewportControl.ApplyHeightBasedCharacterGeometry`; do not switch character layers back to `Stretch=Uniform`, because wide sprites such as `W`'s first emote will render too small compared with AO2.

After height-based scaling, character layers are horizontally centered against the 256x192 canvas before packet offsets are applied. This matches AO2's centered `AnimationLayer` layout and prevents wide sprites from drifting right after they are scaled to viewport height.

Background folder resolution follows AO2 mount priority. `Background.RefreshCache()` and targeted `TryUpsertBackgroundInCache()` keep the first matching background name from `Globals.BaseFolders`; lower-priority duplicate mount folders must not replace the effective background during a targeted refresh.

### Multi-Pass Blurring (Unresolved)
Before the viewport-specific decode hint, viewport images rendered from WebP/APNG assets were decoded through two resize passes:
1. The shared animation decoder capped the decoded frame at `MaxAnimatedPreviewDimension = 360` (see `Ao2AnimationPreview.cs`) — originally a UI preview thumbnail safeguard.
2. WPF `Image` scaled the 360px decode down to the 256x192 display canvas.

AO2 uses a single bilinear pass from native resolution to display size. The two-pass path is the likely cause of the mild extra softening visible compared to AO2.

**Current optimization.** The viewport now passes the target native bounds into `SetAnimatedImage`, `SetPlacedImage`, and the shared animation/static loaders. GIFs keep full frame count for preanimation timing parity, but are scaled to the viewport target dimension during decode. WebP/APNG paths use the same target bound instead of the preview thumbnail bound. Static images use `DecodePixelWidth` where possible.

**Do not retry this broken approach.** Bypassing the resize path entirely by passing `maxDimension: 0` to the old Magick.NET decoder caused corrupt pixel data at native resolution: `GetPixels().ToByteArray(PixelMapping.BGRA)` returned garbage when no `Resize` call normalized Magick's internal pixel format. The always-resize workaround (resizing even when `targetWidth == sourceWidth && targetHeight == sourceHeight`) did not fix this because Magick elided same-dimension resizes as a no-op. The broken state manifested as a brown square BG and invisible overlays. WebP decoding now uses SkiaSharp (`TryDecodeAnimationFramesWithSkia` / `TryDecodeWebPFirstFrameWithSkia`) which does not have this issue.

**Current cache layer.** `Ao2AnimationPreview` keeps a bounded decoded-animation cache keyed by path, last-write time, and target dimension. This does not remove the first decode cost for a new asset, but it prevents repeated stalls when the same expensive background/position asset is used again after another position temporarily replaced it.
